# ADR-003: YAML Frontmatter Schema for Notes

## Context

The explorer needs structured metadata (cluster, modality, beamline, tags) to support filtering, metadata panels, and IA mapping. Currently, none of the 200+ note files have YAML frontmatter — they start directly with markdown headings. We need a schema that is expressive enough for the explorer's features but simple enough for contributors to adopt incrementally. We also need controlled vocabularies to ensure consistent filtering (e.g., "tomography" vs "tomo" vs "CT" should resolve to one canonical value).

## Decision

We adopt the YAML frontmatter schema defined in `docs/03_implementation/data_contracts.md`. Required fields are `title`, `cluster`, `tags`, `modality`, and `beamline`. Controlled vocabularies are enforced for `cluster` (3 values), `modality` (7 values), and `beamline` (18 values). The explorer validates frontmatter at runtime: invalid vocabulary values produce warnings (logged to console), not errors. Notes without frontmatter load with inferred defaults (title from filename, cluster from folder, empty tags).

## Consequences

### Positive
- Enables tag-based filtering, metadata panels, and beamline badges.
- Controlled vocabularies prevent fragmentation (e.g., no duplicate tags).
- Graceful degradation means the explorer works immediately, before any notes are annotated.
- Schema is simple enough for non-developers to add frontmatter.

### Negative / Trade-offs
- Existing 200+ notes need progressive annotation — this is a sustained effort.
- Contributors must learn the controlled vocabulary values.
- Schema changes require updating `data_contracts.md` and potentially `explorer/lib/notes.py`.

### Follow-ups
- Create a linter script (`scripts/validate_frontmatter.py`) to check notes against the schema.
- Prioritize annotating notes in the "Explore the Science" cluster first (highest filtering value).
- Document the schema in `CONTRIBUTING.md` so new contributors know the format.

## Alternatives Considered

- **Free-form tags only (no controlled vocabulary)**: Simpler to adopt but leads to inconsistent filtering (e.g., "XRF" vs "xrf_microscopy" vs "x-ray fluorescence"). Rejected because filtering UX depends on consistent values.
- **Metadata in a separate YAML file per folder**: Keeps notes clean but adds a second file to maintain per note. Rejected because frontmatter is the markdown ecosystem standard and keeps metadata co-located with content.
