# ADR-006: Dual SemVer Streams (notes vs explorer)

## Context

The repository contains two distinct artifacts: (1) the note content (folders 01–09) and (2) the explorer application (`explorer/`). These have different change velocities — notes may be updated daily as new papers are reviewed or tools are documented, while the explorer changes in discrete feature phases (A–E). A single version number would conflate content updates with application releases, making it hard to communicate what changed. We need a versioning scheme that reflects the independence of content and application.

## Decision

We adopt two independent SemVer streams:

- **`notes-vX.Y.Z`** — versions the content in the note folders. Incremented when notes are added, substantially revised, or reorganized.
- **`explorer-vX.Y.Z`** — versions the explorer application. Incremented per phase delivery (A → 0.1.0, B → 0.2.0, etc.).

Each stream has its own git tag, release note in `docs/05_release/release_notes/`, and entry in `CHANGELOG.md`. The CHANGELOG uses separate sections for each stream.

## Consequences

### Positive
- Clear communication: "explorer-v0.2.0 adds cluster pages" vs "notes-v0.1.0 baselines existing content."
- Independent release cadences — a typo fix in a note doesn't require an explorer version bump.
- Git tags enable precise rollback to any content or application state.
- Release notes are organized by artifact, making it easy to see what changed in each.

### Negative / Trade-offs
- Two version streams add mild cognitive overhead for contributors.
- Some changes span both (e.g., adding frontmatter to notes to support a new explorer feature). In such cases, both streams are bumped.
- CI/CD (if added later) needs to handle two tag patterns.

### Follow-ups
- Create `notes-v0.1.0` tag at the current content baseline.
- Create `explorer-v0.1.0` tag after Phase A delivery.
- Document the dual versioning scheme in `docs/06_meta/contributing.md`.

## Alternatives Considered

- **Single SemVer**: One version for the entire repo. Rejected because content and application have different velocities, and a version bump for a note edit is misleading about application state.
- **CalVer (calendar versioning)**: Version by date (e.g., 2026.04.08). Rejected because it doesn't convey the magnitude of change (patch vs minor vs major).
- **No formal versioning**: Use git commits only. Rejected because stakeholder communication ("what shipped?") requires named versions with changelogs.
