# ADR-001: Choose Streamlit over Next.js / Docusaurus / Jekyll

## Context

We need a web framework to build an interactive portal over the synchrotron-data-analysis-notes repository. The portal must render markdown notes at runtime, support tag-based filtering, and present a custom visual design aligned with ANL/APS standards. The development team has strong Python expertise from the scientific computing domain but limited frontend JavaScript experience. Rapid iteration speed matters — we have a 12-week delivery timeline. The framework must support hot-reload development, be deployable with minimal ops overhead, and integrate naturally with the Python data science ecosystem (pandas, markdown parsers, Pygments).

## Decision

We will use **Streamlit** as the application framework for eBERlight Explorer. Streamlit's Python-native API, built-in markdown rendering, multi-page app support, and zero-config deployment (via Streamlit Community Cloud or a simple `streamlit run` command) align with our team's skills and timeline. Custom visual design will be achieved through `st.markdown(unsafe_allow_html=True)` and a custom CSS file.

## Consequences

### Positive
- Python-only stack — no Node.js build toolchain, no JS bundling.
- Native Jupyter notebook rendering capability for future EDA integration.
- Hot-reload development loop (sub-second on file save).
- Deployment to Streamlit Community Cloud requires zero infrastructure.
- The scientific Python community already knows Streamlit.

### Negative / Trade-offs
- CSS customization is limited to injected styles via `unsafe_allow_html`. Complex layouts require workarounds.
- No native SSR/SSG — every page load runs Python server-side. Performance depends on caching.
- The Streamlit component model is less flexible than React/Vue for custom interactive widgets.
- ANL design system compliance requires extra effort to override Streamlit's default theme.

### Follow-ups
- Create `explorer/.streamlit/config.toml` with design-system theme tokens.
- Establish a `components/` pattern for reusable HTML/CSS rendered via `st.markdown`.
- Benchmark landing page TTI against the < 2s target (NFR-001).

## Alternatives Considered

- **Next.js**: Full design freedom, SSR/SSG, React component ecosystem. Rejected because the team lacks React/TypeScript expertise, and the JS build toolchain adds ops complexity disproportionate to the project scope.
- **Docusaurus**: Purpose-built for documentation sites with markdown support. Rejected because it doesn't support runtime data filtering, tag-based navigation, or Python-based note parsing — it's a static site generator, not an interactive portal.
- **Jekyll**: Simple static site generator with GitHub Pages integration. Rejected for the same reasons as Docusaurus, plus Ruby dependency and slower iteration speed.
