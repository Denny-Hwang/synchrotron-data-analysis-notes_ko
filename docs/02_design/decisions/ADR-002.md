# ADR-002: Notes Remain Single Source of Truth

## Context

The repository contains 200+ markdown files across 9 note folders documenting synchrotron data analysis knowledge. The explorer needs to present this content. We must decide whether the explorer maintains its own content database (e.g., YAML catalogs, JSON indices, or a SQLite database) or reads directly from the note files at runtime. Content duplication introduces synchronization risk — if a note is updated but the explorer's copy is not, users see stale information. Conversely, reading files at runtime adds I/O overhead and makes the explorer dependent on the file system layout.

## Decision

The explorer SHALL read note files directly from the 8 note folders (plus 09\_noise\_catalog) at runtime. No content duplication is permitted. The explorer may build in-memory indices at startup (e.g., a list of notes with parsed frontmatter), but these indices are ephemeral and regenerated on each app restart. Any edit to a note file is automatically reflected the next time the explorer loads.

## Consequences

### Positive
- Zero synchronization overhead — one source of truth, no stale copies.
- Contributors only need to edit note files; no secondary update step.
- The explorer is always current with the repository HEAD.
- Reduces repository size — no duplicate content in `explorer/data/`.

### Negative / Trade-offs
- Startup time increases with corpus size (currently ~250 files, manageable).
- The explorer depends on the file system layout of the note folders. Renaming or moving folders requires updating the IA mapping in `explorer/lib/ia.py`.
- Full-text search requires building an index at startup rather than querying a pre-built database.

### Follow-ups
- Implement `explorer/lib/notes.py` with `@st.cache_data` to avoid re-reading files on every page navigation.
- Monitor startup time as the corpus grows; consider a build-time index if it exceeds 10s.
- The existing `eberlight-explorer/data/` YAML catalogs (from the prior explorer) will not be used by the new explorer.

## Alternatives Considered

- **Pre-built YAML/JSON index**: The existing `eberlight-explorer/` uses 6 YAML catalogs (`content_index.yaml`, etc.) as an intermediate data layer. Rejected because it requires a manual `build_index.py` step and introduces sync risk.
- **SQLite database**: Full-text search and structured queries. Rejected as over-engineering for a ~250-file corpus with simple filtering needs.
