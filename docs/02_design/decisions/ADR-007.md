# ADR-007: Static site mirror of Streamlit explorer for GitHub Pages

## Context

The eBERlight Explorer (`explorer/`) is a Streamlit app. Streamlit requires
a Python runtime to serve requests and therefore cannot run on GitHub Pages,
which only hosts static files. Per ADR-001 we chose Streamlit for its fit
with the team's Python skills and for fast iteration; we do not want to
revisit that.

At the same time, we do have a GitHub Pages deployment — until now it only
published the three static HTML design wireframes in
`docs/02_design/wireframes/html/`. The Pages site therefore gave a
misleading impression of the product because it showed only the design
mockups and none of the actual content.

We want readers without a local Python environment to browse the same
content through the same information architecture the Streamlit app
presents, and we want that mirror to stay in lockstep with Streamlit as the
app evolves.

## Decision

We build a second, read-only presentation of the explorer as a static
HTML site and publish it to GitHub Pages on every push to `main`.

Mechanics:

- A generator script `scripts/build_static_site.py` at the repo root reads
  the source-of-truth note folders and emits HTML into `site/`.
- The generator **imports** the Streamlit app's internal modules
  (`explorer/lib/ia.py` for the IA mapping, `explorer/lib/notes.py` for
  the note parser) rather than re-implementing them. The IA and the `Note`
  dataclass have exactly one definition, shared between Streamlit and the
  static site.
- The generator also inlines the Streamlit app's CSS
  (`explorer/assets/styles.css`) into the produced site so visual tokens
  cannot drift.
- `.github/workflows/pages.yml` runs the generator and deploys `site/` to
  GitHub Pages. The workflow is retriggered whenever a note folder,
  `explorer/**`, `scripts/build_static_site.py`, the wireframes, or the
  workflow itself changes.
- A new invariant in `CLAUDE.md` (#9) and a sync contract document
  (`docs/03_implementation/github_pages_sync.md`, IMPL-002) require that
  any change to `explorer/` ships with the matching change to
  `scripts/build_static_site.py` in the same PR.

The generated `site/` directory is git-ignored and regenerated on every
build; it is never hand-edited.

## Consequences

### Positive

- Readers without Python still see all 176+ notes through the 3-cluster IA
  at the canonical Pages URL, not just the 3 design wireframes.
- Shared imports (`explorer/lib/ia.py`, `explorer/lib/notes.py`) eliminate
  drift for IA and parsing — these are defined once and consumed twice.
- Shared CSS inlined at build time means visual tokens (ADR-005, DS-001)
  stay identical between the two surfaces.
- The Streamlit app remains the canonical interactive client; we did not
  move off Streamlit (ADR-001 stands).
- The static site preserves all existing wireframe URLs under
  `site/wireframes/`, so nothing that previously worked on Pages breaks.

### Negative / Trade-offs

- Streamlit-only features that depend on a running Python process (form
  state, live recomputation, uploads) cannot be mirrored and must be
  treated as Streamlit-only or avoided in core navigation.
- The generator replicates the HTML shells of the `card`, `header`,
  `footer`, `breadcrumb`, and `note_view` components. Changing a component
  therefore requires updating two places — the Streamlit component and the
  corresponding generator function. The sync contract (IMPL-002) documents
  this mapping and puts the responsibility on the PR author.
- Every push to `main` that touches a note now also triggers a full site
  rebuild and Pages deploy (~seconds; acceptable).
- The wireframes-only workflow is replaced, which subtly changes the
  trigger paths. We accept this because the new workflow is a strict
  superset of the old one.

### Follow-ups

- Add a CI smoke step that imports the generator and asserts `site/`
  contains `index.html`, `clusters/*.html`, and at least one note page, so
  silent regressions are caught.
- Consider adding a simple client-side search (Lunr.js or Pagefind) in a
  later iteration to match the Streamlit app's search UX.
- Once the Streamlit app gains features that don't mirror (e.g. interactive
  filtering), document the gap explicitly in IMPL-002.

## Alternatives Considered

- **Keep publishing only the design wireframes.** Rejected — the
  landing experience on Pages does not represent the product and hides the
  176 notes.
- **Port the explorer to `stlite` (Streamlit running in WebAssembly).**
  Rejected — adds significant bundle size (tens of MB of Pyodide), long
  cold start, and imperfect Streamlit compatibility for a read-only use
  case where static HTML is strictly simpler.
- **Deploy the Streamlit app itself to Streamlit Community Cloud and link
  from Pages.** Rejected as the primary mirror — introduces a separate
  service with its own availability and sleep behavior. Still an option
  alongside Pages if we later want interactivity on the hosted build, but
  Pages should remain the read-only mirror that always works.
- **Generate the mirror with a generic SSG (MkDocs, Docusaurus).** Rejected
  — we would either lose the ANL-aligned DS-001 tokens or have to re-encode
  them in a theme. Sharing Streamlit's own CSS via a small Python generator
  is the lowest-drift path.
