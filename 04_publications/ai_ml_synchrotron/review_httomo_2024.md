## 한 줄 요약

HTTomo는 고처리량 방사광 빔라인을 위해 설계된 모듈러, GPU 가속 토모그래피 처리 파이프라인이다. 현대 고속 검출기가 제기하는 데이터 속도 문제를, 전처리, 재구성, 후처리 단계를 플러그인 아키텍처로 연결하기 위한 YAML 기반 구성 시스템을 제공함으로써 해결한다. HTTomo는 TB 규모 데이터셋의 거의 실시간 처리를 달성하며, 핵심 프레임워크를 수정하지 않고 새로운 알고리즘을 플러그인으로 쉽게 통합하도록 설계되었다.

---

## 배경 및 동기

현대 방사광 토모그래피 빔라인은 기존 처리 파이프라인을 압도하는 속도로 데이터를 생성한다:

- **데이터 속도**: 업그레이드된 시설(Diamond-II, APS-U, ESRF-EBS)의 고속 검출기는 다중 GB/s 속도로 데이터를 생성하며, 실험당 TB 규모 데이터셋이 발생한다.
- **처리 병목**: 순차적, 모놀리식 처리 스크립트는 속도를 따라잡을 수 없어 적체를 일으키고 과학적 피드백을 지연시킨다.
- **알고리즘 다양성**: 서로 다른 실험은 서로 다른 전처리, 재구성, 후처리 단계 조합을 요구하며, 이는 유연한 파이프라인 구성을 필요로 한다.
- **재현성**: 임시(ad hoc) 스크립트는 재현, 공유, 버전 관리가 어렵다.

HTTomo는 GPU 가속과 선언적 구성 접근법을 활용하는 모듈러 프레임워크로 이러한 문제를 다룬다.

## 방법

### 아키텍처

| 항목 | 세부사항 |
|------|---------|
| **프레임워크** | GPU 가속 백엔드를 가진 Python 기반 파이프라인 오케스트레이터 |
| **구성** | 처리 체인을 지정하는 YAML 기반 워크플로우 정의 |
| **플러그인 시스템** | 각 처리 단계(전처리, 재구성, 후처리)를 위한 모듈러 플러그인 |
| **GPU 백엔드** | CuPy, TomoPy (GPU), 커스텀 CUDA 커널 |
| **I/O** | HDF5/NeXus 입력; 대형 데이터셋의 메모리 효율적 처리를 위한 청크 기반 처리 |

### 파이프라인 설계

HTTomo는 처리 워크플로우를 정의하기 위해 YAML 구성 파일을 사용한다:

```yaml
- method: normalize
  module_path: httomo.methods.preprocessing
  parameters:
    cutoff: 10

- method: find_center_vo
  module_path: httomo.methods.misc

- method: recon
  module_path: httomo.methods.reconstruction
  parameters:
    algorithm: FBP_CUDA
    center: auto
```

**주요 설계 원칙**:

- **플러그인 아키텍처**: 각 처리 단계는 표준화된 입력/출력 인터페이스를 가진 자체 완결된 플러그인이다. 새로운 알고리즘은 핵심 프레임워크를 수정하지 않고 추가 가능.
- **청크 기반 처리**: 대형 데이터셋이 GPU 메모리에 맞는 청크로 분할되어, 임의로 큰 부피의 처리를 가능하게 함.
- **자동 데이터 전송**: 프레임워크가 CPU-GPU 데이터 전송을 관리하여 플러그인 코드의 보일러플레이트를 최소화.
- **MPI 병렬성**: 클러스터 배포를 위한 MPI를 통한 다중 GPU 및 노드 간 분산 처리.

### 파이프라인

```
원시 프로젝션 데이터 (HDF5/NeXus)
  --> YAML 워크플로우 구성
  --> HTTomo 파이프라인 오케스트레이터
      --> 전처리 플러그인 (정규화, 링 제거, 위상 복원)
      --> 재구성 플러그인 (FBP, 반복, AI 기반)
      --> 후처리 플러그인 (분할, 정량화)
  --> 재구성된 부피 (HDF5)
```

## 강점

- **모듈러 및 확장 가능**: 플러그인 아키텍처가 프레임워크 변경 없이 새로운 알고리즘(AI/ML 방법 포함)의 쉬운 추가를 허용.
- **GPU 가속**: CuPy 및 CUDA 백엔드를 활용하여 현대 데이터 속도와 호환되는 고처리량 처리.
- **선언적 구성**: YAML 기반 워크플로우 정의는 사람이 읽을 수 있고, 버전 관리 가능하며, 본질적으로 재현 가능.
- **확장성**: 청크 기반 처리 및 MPI 지원이 다중 GPU 클러스터에 걸친 임의로 큰 데이터셋의 처리를 가능하게 함.
- **활발한 개발**: 정기적 업데이트 및 커뮤니티 기여로 Diamond Light Source가 유지 관리.
- **시설에서 검증됨**: Diamond Light Source 빔라인에 배포 및 검증.

---

## 한계 및 격차

- **Diamond 중심 기본값**: 일반화 가능하지만, 기본 플러그인 및 데이터 형식은 Diamond Light Source 관습에 최적화되어 있어 APS 데이터 형식 적응에는 구성이 필요.
- **제한된 AI/ML 통합**: 현재 플러그인 라이브러리는 고전적 알고리즘에 초점; 딥러닝 재구성 및 노이즈 제거 플러그인은 아직 개발 중.
- **문서 격차**: 일부 고급 기능(커스텀 플러그인 개발, MPI 구성)에 대한 문서가 제한적.
- **단일 측정 기법 초점**: 토모그래피용 설계; 다른 방사광 기법(타이코그래피, XRF)으로의 확장은 상당한 작업이 필요.

## APS BER 프로그램과의 관련성

HTTomo는 APS-U 데이터 처리 문제에 직접적으로 적용 가능하다:

- **APS-U 데이터 속도**: GPU 가속, 청크 기반 아키텍처는 APS-U 빔라인에서 예상되는 정확히 그러한 TB 규모 데이터 속도를 위해 설계됨.
- **파이프라인 표준화**: YAML 기반 구성이 APS 빔라인에 걸쳐 토모그래피 처리를 표준화하여 재현성을 개선하고 빔라인 과학자 부담을 감소시킬 수 있음.
- **AI 플러그인 통합**: 플러그인 아키텍처는 BER 프로그램 AI/ML 방법(TomoGAN, 확산 모델)을 프로덕션 파이프라인 내 처리 단계로 배포하기 위한 자연스러운 프레임워크 제공.
- **Bluesky/Tiled 통합**: HTTomo의 모듈러 설계는 처리 백엔드로서 Bluesky 생태계와의 통합과 호환됨.
- **우선순위**: **높음** -- 성숙하고 시설에서 검증된 프레임워크로 APS-U 데이터 처리량 문제를 직접 다룸.

---

## 실행 가능한 시사점

1. **APS-U 배포를 위한 평가**: 대표적 APS 데이터셋에서 HTTomo를 벤치마크하고 기존 APS 처리 파이프라인과 비교.
2. **APS 특화 플러그인 개발**: APS 데이터 형식, 빔라인 특화 전처리, BER AI/ML 방법을 위한 플러그인 생성.
3. **Bluesky/Tiled와 통합**: Bluesky에서 HTTomo 파이프라인을 트리거하고 결과를 Tiled 카탈로그에 저장하기 위한 커넥터 개발.
4. **AI/ML 플러그인 추가**: TomoGAN, Noise2Void, 기타 BER 프로그램 방법을 프로덕션 배포를 위한 HTTomo 플러그인으로 패키징.
5. **TomocuPy와의 벤치마크**: 도구 선택을 위해 동일한 데이터셋에서 TomocuPy와 처리량 및 재구성 품질 비교.

## 참고 및 논의

HTTomo는 전 세계 업그레이드된 방사광 시설 전반에서 공통적인 고처리량 토모그래피 처리 문제에 대한 성숙한 접근법을 대표한다. 그 플러그인 아키텍처는 본 컬렉션에서 리뷰된 AI/ML 방법을 배포하기 위한 자연스러운 후보가 되게 한다. `05_tools_and_code/httomo/README.md`의 동반 도구 항목이 설치 및 사용 세부 사항을 제공한다.
