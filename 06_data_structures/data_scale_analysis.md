# 데이터 규모 분석: APS-U 전후 비교

## 개요

Advanced Photon Source Upgrade (APS-U)는 X선 밝기의 혁신적인 도약을 나타내며,
기존 APS보다 최대 500배 밝은 빔을 제공합니다. 이러한 개선은 더 빠른 수집, 더 높은 공간 분해능,
그리고 새로운 실험 모달리티를 가능하게 합니다 -- 하지만 동시에 데이터 생성 속도와 총 볼륨의
극적인 증가를 유발하여 데이터의 관리, 저장, 처리 방식을 근본적으로 변화시킵니다.

이 문서는 BER 프로그램과 광범위한 APS 사용자 커뮤니티가 직면한 데이터 규모 과제를 정량화합니다.

## 데이터 생성 속도 대 처리 속도

데이터 생성이 실시간 처리 및 분석 능력을 초과한다는 점이 핵심적인 과제입니다:

| 단계 | APS-U 이전 속도 | APS-U 이후 속도 | 격차 |
|------|----------------|-----------------|------|
| 수집 | 0.5--1 GB/s | 5--20 GB/s | -- |
| 네트워크 전송 (빔라인 -> 저장소) | 10 Gb/s | 100 Gb/s | 충분 |
| 디스크 쓰기 (GPFS) | 5 GB/s | 20 GB/s | 한계적 |
| 단층촬영 재구성 (CPU) | 0.3 GB/s | 0.3 GB/s | 17--67배 느림 |
| 단층촬영 재구성 (GPU) | 5 GB/s | 5 GB/s | 대부분 충분 |
| AI/ML 추론 (GPU당) | 2 GB/s | 2 GB/s | 다중 GPU 필요 |
| 전체 파이프라인 처리량 | 0.2 GB/s | 0.5--2 GB/s | 10--40배 격차 |

## BER 프로그램에 대한 시사점

### 데이터 관리

- **자동화된 파이프라인**은 더 이상 선택사항이 아닙니다 -- 수동 처리는 속도를 맞출 수 없습니다
- 빔라인에서의 **엣지 컴퓨팅** (GPU, FPGA)이 실시간 축소에 필요합니다
- 일부 원시 데이터에 대해 **손실 압축**이 필요할 수 있습니다 (ZFP, SZ 압축기)
- **데이터 생명주기 정책**이 보존 계층을 정의해야 합니다 (핫 / 웜 / 콜드 / 아카이브)

### AI/ML 기회

데이터 규모 과제는 AI/ML이 가장 큰 가치를 제공하는 정확한 영역입니다:

| 과제 | AI/ML 솔루션 |
|------|-------------|
| 실시간 품질 평가 | CNN 기반 이상 탐지 |
| 소스에서의 데이터 축소 | 오토인코더 압축 |
| 더 빠른 재구성 | 신경망 사전정보 (PtychoNN) |
| 자동 특징 추출 | 의미론적 분할 |
| 실험 조향 | 스캔 최적화를 위한 강화학습 |

### 컴퓨팅 요구사항

단일 빔라인의 APS-U 이후 처리 요구사항:

| 워크로드 | APS-U 이전 | APS-U 이후 |
|----------|-----------|------------|
| CPU 코어 | 32--128 | 256--1024 |
| GPU 카드 | 0--1 | 4--16 |
| RAM | 128--512 GB | 1--4 TB |
| 로컬 NVMe | 2--10 TB | 20--100 TB |

## 요약

APS-U 업그레이드는 모달리티에 따라 BER 프로그램 데이터 볼륨을 **50--200배** 증가시킵니다.
시설 전체의 연간 저장 필요량은 약 1 PB에서 60 PB로 증가합니다. 이 규모는 자동화된 데이터
파이프라인, GPU 가속 처리, 엣지에서의 지능적 데이터 축소, 그리고 과학적 발견의 속도를
유지하기 위한 AI/ML 기반 분석을 요구합니다.
