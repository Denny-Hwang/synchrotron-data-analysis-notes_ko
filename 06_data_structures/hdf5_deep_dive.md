# HDF5 심층 분석: 내부 구조, 성능, 한계점

## 개요

이 문서는 [HDF5 README](hdf5_structure/README.md)에서 다룬 HDF5 기본 내용을 넘어,
APS-U 규모의 데이터(빔라인당 5--20 GB/s)를 처리하는 데 필수적인 내부 메커니즘,
성능 튜닝 전략, 알려진 한계점을 심도 있게 살펴본다. 이러한 세부 사항을 이해하는
것은 차세대 검출기의 속도에 대응할 수 있는 데이터 파이프라인을 설계하는 데 필수적이다.

## HDF5 내부 아키텍처

### 디스크 상의 파일 구조

HDF5 파일은 단순한 플랫 바이트 스트림이 아니다. 내부적으로 다음과 같은 구성 요소로 이루어져 있다:

| 구성 요소 | 역할 |
|-----------|------|
| **Superblock** | 파일 서명, 버전, 루트 그룹 주소; 파일 오프셋 0에 위치 |
| **B-tree** | 그룹의 하위 항목 및 청크 데이터셋 저장소 색인 |
| **Object headers** | 메타데이터(데이터 공간, 데이터 타입, 필터, 속성) 저장 |
| **Data chunks** | 실제 배열 데이터로, 독립적으로 접근 가능한 단위로 저장됨 |
| **Free-space manager** | 삭제 후 재사용 가능한 미사용 영역 추적 |

B-tree 인덱싱을 통해 임의의 청크에 대한 랜덤 접근이 O(N)이 아닌 O(log N)으로 이루어진다.
HDF5 1.10+ 버전에서는 추가적인 청크 인덱스 유형이 도입되었다:

| 인덱스 유형 | 복잡도 | 사용 사례 |
|-----------|-----------|----------|
| B-tree v2 | O(log N) | 확장 가능한 데이터셋의 기본값 |
| Single chunk | O(1) | 데이터셋이 하나의 청크에 들어맞는 경우 |
| Fixed array | O(1) | 고정 크기 데이터셋, 차원이 알려진 경우 |
| Extensible array | O(1) 상각 | 하나의 무제한 차원 (APS에서 가장 일반적) |

### Virtual Datasets (VDS)

Virtual Datasets은 데이터를 복사하지 않고도 하나의 논리적 데이터셋이
여러 물리적 HDF5 파일에 걸쳐 존재할 수 있게 해준다. 이는 EIGER와 같은
검출기가 여러 파일에 걸쳐 데이터를 기록하는 APS에서 매우 중요하다:

```
# EIGER는 여러 데이터 파일을 기록한다:
scan_001_master.h5      # VDS 매핑 + 메타데이터
scan_001_data_000001.h5 # 프레임 1-1000
scan_001_data_000002.h5 # 프레임 1001-2000
...
```

```python
import h5py

# 마스터 파일을 통해 투명하게 읽기가 가능하다
with h5py.File("scan_001_master.h5", "r") as f:
    frame_500 = f["/entry/data/data"][500]  # data_000001.h5로 자동 해석됨
```

VDS의 장점:
- 데이터 중복 없음
- 개별 파일이 파일시스템 크기 제한 이내로 유지됨
- 별도 파일에 병렬 쓰기 후, 마스터를 통한 통합 읽기 가능

## SWMR (Single Writer Multiple Reader)

### 개념

SWMR 모드는 HDF5 1.10에서 도입되었으며, **하나의 프로세스가 데이터를 쓰는** 동안
**여러 프로세스가 동시에 읽을 수** 있게 해준다 -- 파일을 닫거나 다시 열 필요가 없다.
이는 APS에서 실시간 데이터 스트리밍의 기반이 된다.

### 동작 원리

```
Writer 프로세스                   Reader 프로세스
─────────────────                ──────────────────────
파일 열기 (SWMR 쓰기 모드)        파일 열기 (SWMR 읽기 모드)
데이터셋 생성 (SWMR 전)
SWMR 모드 활성화
  │                              │
  ├─ 청크 추가 ──────────────────┤─ H5Drefresh()
  ├─ H5Dflush()                  ├─ 새 데이터 읽기
  ├─ 청크 추가 ──────────────────┤─ H5Drefresh()
  ├─ H5Dflush()                  ├─ 새 데이터 읽기
  │                              │
파일 닫기                         파일 닫기
```

### APS에서의 실용적 활용

```python
import h5py
import numpy as np

# Writer (areaDetector IOC 또는 수집 스크립트)
f = h5py.File("live_scan.h5", "w", libver="latest")
dset = f.create_dataset("data", shape=(0, 2048, 2048),
                        maxshape=(None, 2048, 2048),
                        chunks=(1, 2048, 2048),
                        dtype="uint16")
f.swmr_mode = True  # SWMR 활성화 -- 이후 새 데이터셋 생성 불가

for frame in detector_stream():
    dset.resize(dset.shape[0] + 1, axis=0)
    dset[-1] = frame
    dset.flush()

f.close()
```

```python
# Reader (모니터링 / 실시간 분석)
f = h5py.File("live_scan.h5", "r", libver="latest", swmr=True)
dset = f["data"]

while True:
    dset.refresh()
    n_frames = dset.shape[0]
    if n_frames > last_seen:
        new_data = dset[last_seen:n_frames]
        process(new_data)
        last_seen = n_frames
```

### SWMR의 한계점

| 한계점 | 영향 | 우회 방법 |
|-----------|--------|-----------|
| SWMR 활성화 후 새 데이터셋 생성 불가 | 모든 데이터셋을 사전 생성해야 함 | maxshape=None으로 플레이스홀더 데이터셋 생성 |
| SWMR 중 새 속성 추가 불가 | 스캔 중 메타데이터 추가 불가 | 별도 파일에 메타데이터 기록 또는 SWMR 종료 후 기록 |
| SWMR 중 그룹 생성 불가 | 계층 구조 추가 불가 | 전체 그룹 트리를 사전 생성 |
| 단일 writer만 가능 | 병렬 쓰기 불가 | 별도 파일 + VDS를 사용하여 다중 writer 구현 |
| Flush 지연 시간 ~1-10 ms | 완전한 무지연이 아님 | 10 kHz 미만의 프레임 레이트에서는 허용 가능 |

## 청킹 전략 심층 분석

### 청킹이 중요한 이유

APS-U 데이터 속도에서 청킹은 HDF5 설정 중 **성능에 가장 큰 영향을 미치는** 선택이다.
잘못된 청크 형태는 10-100배의 성능 저하를 유발할 수 있다.

### 접근 패턴 정렬

형태가 `(n_proj, n_row, n_col)`인 토모그래피 데이터셋의 경우:

| 접근 패턴 | 최적 청크 형태 | 사용 사례 |
|---------------|-------------------|----------|
| 프로젝션 단위 | `(1, n_row, n_col)` | 수집, 플랫 필드 보정 |
| 사이노그램 (행) | `(n_proj, 1, n_col)` | 토모그래피 재구성 |
| 부분 볼륨 | `(16, 256, 256)` | 3D 렌더링, 분할(세그멘테이션) |

**근본적인 충돌**: 수집 과정에서는 프로젝션 단위로 기록하지만, 재구성 시에는
사이노그램 단위로 읽어야 한다. 두 가지 모두에 최적인 단일 청크 형태는 존재하지 않는다.

### APS 해결 방법: 2단계 전략

```
1단계: 수집
  └─ 청크 (1, 2048, 2048)로 기록 → 프로젝션 최적화
  └─ ~10 GB/s 쓰기 처리량

2단계: 재청킹 (ALCF에서)
  └─ (1800, 1, 2048)로 재청킹 → 사이노그램 최적화
  └─ 빠른 TomocuPy 재구성 가능
  └─ 또는 Zarr 중간 형식 사용 (data_formats_comparison.md 참조)
```

### 청크 크기 지침

| 청크 크기 | 읽기 오버헤드 | 캐시 효율 | APS 권장 사항 |
|-----------|--------------|-----------------|-------------------|
| < 10 KB | 매우 높음 (B-tree 탐색이 지배적) | 낮음 | 절대 사용하지 말 것 |
| 10 KB -- 100 KB | 높음 | 보통 | 1D 스펙트럼에만 사용 |
| 100 KB -- 1 MB | 낮음 | 양호 | 범용 |
| 1 MB -- 10 MB | 최소 | 매우 양호 | 이미지에 권장 |
| 10 MB -- 100 MB | 최소 | 순차 접근 시 우수 | 대형 프레임, 대량 읽기 |
| > 100 MB | 최소 | 부분 읽기 시 메모리 낭비 | 피할 것 |

**APS 기본값**: 단일 검출기 프레임을 하나의 청크로, 일반적으로 2048x2048x2 바이트 = 8 MB.

## APS 규모에서의 압축 성능

### 벤치마크: 토모그래피 원시 데이터 (uint16, 2048x2048)

| 필터 | 압축비 | 압축 속도 (MB/s) | 압축 해제 속도 (MB/s) | CPU 코어 수 |
|--------|-------|-----------------|-------------------|-----------|
| None | 1.0x | N/A | N/A | 0 |
| LZ4 | 1.8--2.5x | 3500 | 4200 | 1 |
| bitshuffle+LZ4 | 3.5--6.0x | 2800 | 3200 | 1 |
| Blosc (LZ4, shuffle) | 3.0--5.0x | 3000 | 3800 | multi |
| Blosc (Zstd, shuffle) | 4.0--7.0x | 800 | 2500 | multi |
| gzip level 1 | 2.5--3.5x | 250 | 400 | 1 |
| gzip level 6 | 3.5--5.0x | 80 | 400 | 1 |
| SZ (lossy, 1e-3) | 15--40x | 600 | 900 | 1 |
| ZFP (lossy, rate 8) | 10--20x | 1200 | 1400 | 1 |

### 파이프라인 단계별 권장 사항

| 단계 | 필터 | 근거 |
|-------|--------|-----------|
| **수집 (실시간)** | bitshuffle+LZ4 | 가장 빠른 무손실 압축; 검출기 속도에 대응 가능 |
| **전송 (Globus)** | 그대로 유지 | 재압축은 시간 낭비 |
| **처리 결과물** | Blosc-Zstd | 손실 없이 아카이브에 가장 좋은 압축비 |
| **아카이브 (HPSS)** | Blosc-Zstd 또는 SZ | 저장 비용이 지배적; 원시 데이터에는 손실 압축 허용 |
| **공개 데이터셋** | gzip level 4 | 최대 호환성 |

### Direct Chunk Write

검출기가 내부적으로 데이터를 압축하는 경우 (예: Dectris EIGER의 펌웨어 내
bitshuffle+LZ4), HDF5의 `H5DOwrite_chunk` API는 전체 필터 파이프라인을 우회한다:

```
표준 쓰기 경로:
  데이터 → HDF5 필터 파이프라인 (압축) → 디스크
  처리량: ~2 GB/s

Direct chunk write 경로:
  사전 압축된 데이터 → 디스크 (필터 우회)
  처리량: ~8 GB/s (디스크 제한)
```

EIGER areaDetector 드라이버는 이 경로를 사용하여 검출기의 전체 데이터 속도를 유지한다.

## Parallel HDF5 (MPI-IO)

### 아키텍처

Parallel HDF5는 MPI-IO를 사용하여 여러 프로세스가 동일한 파일을 동시에 읽고 쓸 수
있게 해준다. 이는 수백 개의 GPU 노드가 하나의 토모그래피 볼륨을 처리하는 ALCF에서
필수적이다.

```
Node 0: 사이노그램 0-99 읽기     ─┐
Node 1: 사이노그램 100-199 읽기  ─┤── 단일 HDF5 파일
Node 2: 사이노그램 200-299 읽기  ─┤   (MPI-IO 사용)
Node 3: 사이노그램 300-399 읽기  ─┘
```

### ALCF에서의 설정

```python
from mpi4py import MPI
import h5py

comm = MPI.COMM_WORLD
rank = comm.Get_rank()

# 집합적 열기
f = h5py.File("volume.h5", "r", driver="mpio", comm=comm)
dset = f["data"]

# 각 rank가 자신의 슬라이스를 읽음
my_start = rank * chunk_size
my_data = dset[my_start:my_start + chunk_size]
```

### Parallel HDF5의 한계점

| 문제 | 설명 |
|-------|-------------|
| **압축과 병렬 쓰기 불가** | 압축된 청크는 병렬로 쓸 수 없음 (락 경합). 우회 방법: 비압축으로 쓴 후 후처리에서 압축 |
| **메타데이터 작업의 직렬화** | 데이터셋/속성 생성이 모든 rank에 걸쳐 직렬화됨. 해결 방법: 한 rank가 생성하고 나머지는 배리어에서 대기 |
| **Lustre에서의 POSIX 락 오버헤드** | 파일 잠금이 성능을 저하시킬 수 있음. 해결 방법: ALCF에서 `HDF5_USE_FILE_LOCKING=FALSE` 사용 |
| **청크 정렬** | 청크가 Lustre 스트라이프 경계(일반적으로 1 MB)에 맞춰 정렬되어야 함 |
| **Collective I/O vs. Independent I/O** | Collective I/O(조율된 방식)가 훨씬 빠르지만 동기화가 필요 |

### ALCF에서의 성능 (Polaris/Aurora)

| 설정 | 읽기 처리량 | 쓰기 처리량 |
|--------|----------------|-----------------|
| 단일 프로세스 | 2--5 GB/s | 1--3 GB/s |
| 32 MPI ranks, collective | 20--40 GB/s | 10--25 GB/s |
| 256 MPI ranks, collective | 50--100 GB/s | 30--60 GB/s |
| Eagle 파일시스템 최대치 | 총 650 GB/s | 총 650 GB/s |

## HDF5의 알려진 한계점

### 1. 단일 파일 락 경합

HDF5는 기본적으로 POSIX 파일 락을 사용한다. 공유 파일시스템(GPFS, Lustre)에서
여러 프로세스가 동일한 파일에 접근할 때 경합이 발생한다. Parallel HDF5가 조율된
접근에 대해 이를 해결하지만, 비조율적 동시 읽기는 여전히 영향을 받을 수 있다.

### 2. 클라우드 / 오브젝트 스토어 접근

HDF5는 POSIX 파일시스템용으로 설계되었다. 클라우드 오브젝트 스토어(S3, Azure Blob)에서
HDF5에 접근하는 것은 문제가 있다:

| 접근 방식 | 지연 시간 | 사용성 |
|----------|---------|-----------|
| 전체 파일 다운로드 | 대용량 파일의 경우 수 분 | 비실용적 |
| HSDS (HDF5 클라우드 서비스) | 요청당 10--100 ms | 서버 배포 필요 |
| kerchunk (가상 Zarr 뷰) | 첫 스캔은 느리나 이후 빠름 | 읽기 전용, 인덱스 필요 |
| ros3 드라이버 (S3 네이티브) | 청크당 50--200 ms | 제한된 기능 |

**이것이 중요한 이유**: APS 데이터가 하이브리드 클라우드/HPC 전략으로 이동함에 따라,
HDF5의 POSIX 의존성은 Zarr와 같은 클라우드 네이티브 형식에 비해 상당한 장벽이 된다.

### 3. 메타데이터 확장성

HDF5는 모든 메타데이터(속성, 데이터셋 헤더)를 데이터와 동일한 파일에 저장한다.
수백만 개의 작은 데이터셋이 있는 파일의 경우, 메타데이터 작업이 병목이 된다:

| 파일 복잡도 | 메타데이터 열기 시간 | 실질적 영향 |
|----------------|-------------------|-----------------|
| 10개 데이터셋 | < 1 ms | 없음 |
| 1,000개 데이터셋 | 10--50 ms | 허용 가능 |
| 100,000개 데이터셋 | 1--10 s | 문제가 됨 |
| 1,000,000개 데이터셋 | 30--120 s | 대화형 작업에 사용 불가 |

이는 주로 픽셀별 스펙트럼 데이터셋을 가진 XRF 매핑 파일(드문 경우)과
광범위한 처리 이력 체인을 가진 파일에 영향을 미친다.

### 4. 파일 간 네이티브 추가 기능 없음

Zarr와 달리 HDF5는 형식 수준에서 "여러 파일에 걸친 데이터셋"이라는
기본 개념이 없다. VDS가 이를 제공하지만 명시적 설정이 필요하다.
마스터 파일이 유실되면 데이터 파일 간의 관계도 유실된다.

### 5. Python GIL 제약 (h5py)

h5py 라이브러리는 Python의 GIL로 인해 사실상 **단일 스레드**로 동작한다.
동일한 HDF5 파일에 대한 멀티 스레드 읽기는 병렬화되지 않는다. 이는
Python 기반 분석 파이프라인에서 상당한 한계점이다:

| 접근 방식 | 처리량 | 비고 |
|----------|-----------|-------|
| 단일 스레드 h5py | 2--5 GB/s | GIL 제한 |
| 멀티 스레드 h5py | 2--5 GB/s (이득 없음) | GIL이 병렬 읽기를 방해 |
| 멀티 프로세스 h5py | 10--30 GB/s | 각 프로세스가 독립적으로 파일 열기 |
| MPI 병렬 (mpi4py + h5py) | 50--100 GB/s | ALCF에서 최상의 성능 |
| Zarr + Dask (대안) | 30--80 GB/s | GIL 제한 없음 |

### 6. 데이터를 진정으로 삭제할 수 없음

HDF5 파일에서 데이터셋을 삭제해도 링크만 제거될 뿐 -- 파일 크기는 **줄어들지 않는다**.
디스크 상의 공간은 가용으로 표시되지만 회수되지 않는다. 압축하려면 `h5repack`이 필요하다:

```bash
# 120 GB 파일을 리팩하여 공간 회수 -- 전체 사본을 생성함
h5repack -f GZIP=4 input.h5 output.h5
# TB 규모 파일의 경우, 이 작업은 매우 비용이 크다
```

### 7. 스키마 변경

기존 HDF5 파일의 구조를 수정하는 것(새 그룹 추가, 데이터셋 이름 변경)은
파일 재작성을 필요로 한다. 인플레이스 스키마 마이그레이션은 존재하지 않는다.
이는 요구 사항이 캠페인 중간에 변경되는 장기 실험을 복잡하게 만든다.

### 8. VOL (Virtual Object Layer) -- 잠재적 미래 해결책

HDF5 1.12+ 버전에서는 **Virtual Object Layer (VOL)**이 도입되었으며,
이를 통해 HDF5 API가 플러그인을 통해 임의의 스토리지 백엔드와 인터페이스할 수 있다.
이론적으로는 애플리케이션 변경 없이 클라우드 오브젝트 스토어에 대한 투명한 접근이
가능해질 수 있다. 그러나 S3 및 기타 클라우드 백엔드를 위한 VOL 커넥터는
아직 프로덕션 수준에 도달하지 못했다.

## APS-U 규모를 위한 모범 사례

### 수집 단계

1. 가능한 경우 알려진 최대 크기로 **데이터셋을 사전 할당**한다
2. 내장 압축 기능이 있는 검출기에는 **direct chunk write를 사용**한다
3. 실시간 모니터링을 위해 **SWMR을 활성화**한다
4. 프로젝션 지향 접근을 위해 **프레임당 하나의 청크**를 사용한다
5. EIGER 방식의 다중 파일 구성에서는 **메타데이터 파일을 대용량 데이터와 분리**한다

### 처리 단계

1. Zarr를 중간 형식으로 사용하여 **접근 패턴에 맞게 재청킹**한다
2. ALCF에서 collective I/O와 함께 **Parallel HDF5를 사용**한다
3. Lustre에서 **파일 잠금을 비활성화**한다 (`HDF5_USE_FILE_LOCKING=FALSE`)
4. **청크를 스트라이프 크기에 정렬**한다 (Lustre에서 1 MB)

### 아카이브 단계

1. 최대 호환성을 위해 **gzip으로 리팩**한다
2. 자체 완결적인 NeXus 파일에 **모든 메타데이터를 포함**한다
3. 기탁 전에 **cnxvalidate로 검증**한다
4. 무결성 확인을 위해 **체크섬(SHA-256)을 사용**한다
