# APS-U 데이터 과제: 일일 100+ TB 처리

## 개요

APS 업그레이드(APS-U)는 X선 밝기를 최대 500배 증가시켜 더 빠른 스캔, 더 높은 해상도,
그리고 새로운 실험 방식을 가능하게 한다. 그 결과 빔라인당 **데이터 생성량이
50~200배 증가**하게 된다. 업그레이드된 시설은 모든 빔라인에서 연간 **100 페타바이트 이상**을
생산할 것으로 예상되며, 이는 평균적으로 **일일 270 TB 이상**, 피크 시에는 일일 500+ TB에
달한다.

이 문서에서는 ANL이 현재 이 대규모 데이터 홍수를 어떻게 처리하고 있는지, 무엇이
잘 작동하고, 무엇이 잘 작동하지 않는지, 그리고 어떤 미해결 과제가 남아 있는지를
살펴본다.

## 문제의 규모

### 측정 방식별 데이터 생성 속도

| 측정 방식 | 검출기 | 피크 속도 | 일반적인 스캔 | 일일 볼륨 |
|----------|----------|-----------|-------------|-------------|
| 고속 토모그래피 | EIGER2 X 9M | 9.4 GB/s | 120 GB / 2분 | 10~60 TB |
| 나노-XRF | Vortex ME4 + EIGER | 2.5 GB/s | 80 GB / 3시간 | 2~20 TB |
| 타이코그래피 | EIGER2 X 500K | 1.2 GB/s | 5.2 GB / 10분 | 1~5 TB |
| XPCS | EIGER2 X 4M | 4.0 GB/s | 50 GB / 30분 | 5~30 TB |
| XANES 분광 토모그래피 | EIGER + 단색화장치 | 9.4 GB/s | 400 GB / 1시간 | 5~30 TB |

### 시설 전체 누적 전망

| 지표 | APS-U 이전 (2023) | APS-U 이후 (2025+) | 증가 배율 |
|--------|-------------------|---------------------|--------|
| 가동 빔라인 수 | ~40 | ~70 | 1.75x |
| 빔라인당 일일 평균 | 300 GB | 15 TB | 50x |
| 시설 일일 (피크) | 12 TB | 500+ TB | 40x |
| 연간 원시 데이터 | 720 TB | 100+ PB | 140x |
| 연간 총계 (+ 처리 데이터) | ~1 PB | ~150 PB | 150x |

## 현재 인프라: 데이터 처리 방식

### 데이터 흐름 아키텍처

```
Detector (EIGER, Pilatus 등)
    │
    ├──[areaDetector IOC]──► 빔라인 SSD 캐시 (10-50 TB, NVMe)
    │                            │
    │                            ├──[Globus 100 Gbps]──► ALCF Eagle (100 PB, GPFS)
    │                            │                           │
    │                            │                           ├──► GPU 처리 (Polaris/Aurora)
    │                            │                           │        │
    │                            │                           │        ├──► TomocuPy 재구성
    │                            │                           │        ├──► AI/ML 추론
    │                            │                           │        └──► 결과 → Eagle
    │                            │                           │
    │                            │                           ├──[Globus Flow]──► Petrel (웜 아카이브)
    │                            │                           └──[Globus Flow]──► HPSS 테이프 (콜드 아카이브)
    │                            │
    │                            └──► APS 중앙 GPFS (5 PB, 30일 보존)
    │
    ├──[ZMQ Stream]──► 실시간 모니터 / 엣지 컴퓨팅
    └──[PV Access]──► Bluesky / 라이브 대시보드
```

### 스토리지 계층 상세

#### Tier 0: 빔라인 SSD 캐시

| 항목 | 값 |
|-----------|-------|
| 하드웨어 | NVMe SSD 어레이, Dell/HPE 서버 |
| 용량 | 빔라인당 10~50 TB |
| 대역폭 | 5~10 GB/s (검출기 속도에 대응) |
| 보존 기간 | 수 시간~수 일 (버퍼 전용) |
| 용도 | 검출기 출력 버퍼링, 실시간 처리 지원 |

**장점**: 모든 검출기에 충분한 속도; 지연 시간 최소화를 위해 로컬에 배치.
**단점**: 용량 제한; 지속적으로 데이터를 이동시켜야 함.

#### Tier 1: APS 중앙 GPFS

| 항목 | 값 |
|-----------|-------|
| 하드웨어 | IBM Spectrum Scale (GPFS), HDD + SSD 계층 |
| 용량 | 5 PB |
| 대역폭 | 합산 50 GB/s |
| 보존 기간 | 30일 |
| 용도 | 모든 APS 빔라인의 단기 공유 스토리지 |

**장점**: 모든 빔라인에서 접근 가능한 공유 파일 시스템; 익숙한 POSIX 인터페이스.
**단점**: 30일 보존 기간이 지나면 데이터를 이전하지 않으면 손실됨; 피크 사용 시 경합 발생.

#### Tier 2: ALCF Eagle

| 항목 | 값 |
|-----------|-------|
| 하드웨어 | HPE Cray ClusterStor E1000, Lustre 파일 시스템 |
| 용량 | 100 PB |
| 대역폭 | 합산 650 GB/s |
| 보존 기간 | 90일 (프로젝트 할당 기준) |
| 접근 | Polaris/Aurora 계산 노드에서 직접 접근 |
| 네트워크 | APS 전용 100 Gbps 링크 (Aurora 스트리밍 링크는 테라비트급) |
| 전용 자원 | Polaris 4랙이 APS 시설 통합용으로 예약 |

**장점**: 대규모 대역폭; 슈퍼컴퓨터에서 직접 접근 가능;
Globus를 통한 외부 공유 지원; Aurora로의 테라비트 링크로 스트리밍 가능.
**단점**: 90일 보존 기간; 할당 기반 접근에는 ALCF 프로젝트가 필요.

#### Tier 3: Petrel (웜 아카이브)

| 항목 | 값 |
|-----------|-------|
| 하드웨어 | ALCF의 3 PB 고속 스토리지 |
| 용량 | 프로젝트 할당당 ~100 TB (확장 가능) |
| 접근 | Globus 전송, REST API, 웹 포털 |
| 보존 기간 | 3년 |
| 공유 | ID 기반 ACL, Globus Auth 그룹 |

**장점**: 협력자와의 손쉬운 공유; 웹 브라우저로 탐색 가능; 메타데이터 검색을
위한 검색 인덱스; DOI 랜딩 페이지.
**단점**: 오브젝트 스토어 지연 시간(초 단위); HPC 계산에는 부적합.

#### Tier 4: HPSS 테이프 아카이브

| 항목 | 값 |
|-----------|-------|
| 하드웨어 | IBM 테이프 라이브러리 (LTO-9) |
| 용량 | 500+ PB |
| 접근 | Globus를 통한 배치 리콜 (수 분~수 시간) |
| 보존 기간 | 10+ 년 (원시 데이터), 영구 (출판물) |
| 비용 | ~$20/TB/년 (SSD ~$200/TB/년 대비) |

**장점**: TB당 비용이 매우 낮음; 사실상 무제한 용량;
이중 복사 정책으로 데이터 안전성 보장.
**단점**: 리콜 지연 시간(수 분~수 시간); 순차 접근만 가능;
대화형 분석에는 부적합.

### 데이터 전송 인프라

#### Globus

Globus는 ANL의 **통합 데이터 관리 레이어**로, 모든 시설 간
전송을 처리한다:

| 전송 경로 | 대역폭 | 지연 시간 | 자동화 |
|--------------|-----------|---------|-----------|
| APS → ALCF (Eagle) | 100 Gbps (12.5 GB/s) | 초 | Globus Flows |
| ALCF → Petrel | 내부 40 Gbps | 초 | Globus Flows |
| ALCF → HPSS | 20 Gbps | 분 | Globus Flows |
| APS → NERSC | ESnet 경유 100 Gbps | 초 | 수동 또는 Flows |
| APS → 협력 기관 | 가변 | 가변 | 수동 전송 |

**핵심 기능**: Globus Flows는 데이터 취득 이후의 전체 파이프라인을
자동화한다: 전송 → 처리 → 아카이빙 → 알림. 한 번의 스캔 완료가
전체 체인을 자동으로 트리거한다. Globus는 **단일 캠페인에서 2.9 PB 전송**을
시연한 바 있어, APS-U 수준의 요구에도 확장 가능함이 입증되었다.

#### Nexus Framework (APS-ALCF 통합)

ANL에서 개발한 Nexus 프레임워크는 APS 빔라인에서 ALCF 슈퍼컴퓨터로의
자동 근실시간 스트리밍을 제공한다:

- **8-ID의 XPCS**: 스페클 패턴이 검출기 → ZMQ → Polaris로 스트리밍되어
  상관 함수 계산; 수 초 내에 결과 반환
- **2-BM/32-ID의 토모그래피**: 프로젝션이 Globus를 통해 Polaris로 전송;
  TomocuPy가 GPU에서 재구성 수행; 스캔 완료 후 수 분 내에 3D 볼륨 확보
- **여러 빔라인의 XRF**: 스펙트럼 데이터가 Polaris GPU에서 MAPS를 사용하여
  원소 피팅 처리

## 잘 작동하는 점

### 1. 자동화된 파이프라인 (Globus Flows + Bluesky)

실험 오케스트레이션을 위한 Bluesky RunEngine과 데이터 관리를 위한
Globus Flows의 조합은 완전히 자동화된 파이프라인을 생성한다:

```
스캔 → HDF5 기록 → 전송 → 처리 → 아카이빙 → DOI → 알림
  └──────── 모두 자동화, 사람의 개입 없음 ──────────┘
```

**이점**: 과학자들이 데이터 관리가 아닌 과학에 집중할 수 있다.
**지표**: 스캔 완료에서 처리된 결과까지 중앙값: 5~15분.
**시연 사례**: Polaris에서 완전 자동화된 Laue 미세회절 파이프라인 --
6~12시간 실험 동안 1~2분마다 스캔 처리, 사람의 개입 없이 수행
(MPI+CUDA로 최대 50개 노드에서 병렬화).

### 2. GPU 가속 재구성

ALCF GPU에서의 TomocuPy는 데이터 생성 속도에 맞는 재구성 처리량을
제공한다:

| 작업 | CPU (TomoPy, 128 코어) | GPU (TomocuPy, 4x A100) |
|----------|------------------------|------------------------|
| 2048x2048, 1800 프로젝션 | 15분 | 45초 |
| 4096x4096, 3600 프로젝션 | 2시간 이상 | 8분 |

### 3. SWMR + ZMQ를 통한 라이브 스트리밍

데이터 취득 중 실시간 데이터 품질 평가:
- **SWMR**: 취득이 기록하는 동안 모니터링 프로세스가 HDF5를 읽음
- **ZMQ**: 엣지 컴퓨팅이 수집 중인 프레임을 실시간으로 처리하여 이상 탐지
- **PV Access**: Bluesky 대시보드에서 라이브 썸네일 및 통계 표시

### 4. 계층형 스토리지 모델

핫/웜/콜드/아카이브 계층이 데이터 수명주기에 맞게 비용을 최적화한다:

| 계층 | $/TB/년 | 접근 시간 | 데이터 비율 |
|------|-----------|------------|-------------------|
| SSD 캐시 | $1,000+ | μs | <1% (현재 스캔) |
| GPFS/Eagle | $200 | ms | 5% (활성 처리) |
| Petrel | $50 | 초 | 15% (최근, 공유) |
| HPSS 테이프 | $20 | 분~시간 | 80% (보관) |

## 잘 작동하지 않는 점

### 1. 리청킹(Rechunking) 병목

검출기는 프로젝션 순서(프레임당 하나의 청크)로 데이터를 기록하지만,
재구성에는 사이노그램 순서가 필요하다. 120 GB 토모그래피 데이터셋의
리청킹에는 10~30분이 소요되며 전체 복사본이 생성된다:

| 문제 | 영향 |
|---------|--------|
| 리청킹 중 2배 스토리지 | SSD/GPFS 부담 증가 |
| 10~30분 I/O 바운드 작업 | 재구성 시작 지연 |
| 단일 노드 병목 | HDF5에서 쉽게 병렬화 불가 |

**완화 방안**: 중간 포맷으로 Zarr 사용(병렬 리청킹) 또는
사이노그램을 실시간으로 처리하는 스트리밍 재구성 구현.

### 2. 대규모 메타데이터 관리

연간 수백만 건의 스캔이 이루어지면, 적절한 데이터셋을 찾는 것이
점점 더 어려워진다:

| 과제 | 현재 상태 |
|-----------|--------------|
| 데이터셋 간 검색 | Petrel 검색 + 수동 큐레이션 |
| 출처 추적 | 파일별 NXprocess (일관성 부족) |
| 실험 간 연결 | 수동, PI의 지식에 의존 |
| 샘플 추적 | 대부분 빔라인에서 스프레드시트 기반 |

**필요한 것**: 표준화된 온톨로지와 자동 추출 기능을 갖춘
시설 전체 메타데이터 카탈로그.

### 3. 30일 GPFS 보존 기간

APS 중앙 GPFS의 30일 보존 기간은 해당 기간 내에 데이터를 전송하고
검증해야 함을 의미한다. 피크 부하 시:

- 15개 빔라인 × 15 TB/일 = 일일 225 TB 수집
- 100 Gbps 전송 = 12.5 GB/s = 이론적 최대 1.08 PB/일
- 실질 처리량 (경합 포함): 일일 500~800 TB
- **다수의 빔라인이 동시에 운영되는 캠페인 기간에는 여유가 매우 적음**

### 4. HPSS 리콜 지연

사용자가 보관된 데이터를 재처리해야 할 때, 테이프로부터의 리콜에는
수 분에서 수 시간이 소요될 수 있다:

| 데이터셋 크기 | 리콜 시간 | 영향 |
|-------------|------------|--------|
| 1 GB | 2~5분 | 허용 가능 |
| 100 GB | 15~30분 | 불편함 |
| 10 TB | 2~8시간 | 하루 동안 워크플로우 차단 |

**완화 방안**: 자주 재처리되는 데이터셋을 Petrel에 유지; 사용자 일정에 기반한
예측적 사전 스테이징 구현.

### 5. 클라우드에서의 HDF5

일부 워크플로우가 클라우드 하이브리드 모델(Google Cloud, AWS)로 이동함에 따라,
HDF5의 POSIX 의존성이 제약이 된다:

- S3에서의 효율적인 부분 읽기 불가
- HDF5 파일의 서버리스 처리 불가
- 전체 다운로드 또는 HSDS/Tiled 서버 배포 필요

## 미해결 과제 및 연구 방향

### 1. 엣지에서의 실시간 AI

**목표**: 즉각적인 피드백을 위해 빔라인에서 AI/ML 모델 실행.

| 과제 | 현재 | 목표 |
|-----------|---------|--------|
| 추론 지연 시간 | 100 ms (GPU 서버) | <10 ms (엣지 FPGA/GPU) |
| 모델 배포 | 수동 | 빔라인으로의 자동화된 CI/CD |
| 이상 탐지 | 사후 분석 | 프레임별, 실시간 |
| 실험 조종 | 사람이 결정 | AI가 다음 측정 제안 |

### 2. 원시 데이터의 손실 압축

연간 100+ PB 규모에서는 테이프조차 비용이 크다. 손실 압축은
원시 데이터를 10~50배 줄일 수 있지만 과학적 위험을 수반한다:

| 압축기 | 압축률 | 오차 범위 | APS에서의 현황 |
|-----------|-------|------------|--------------|
| SZ3 | 15~50x | 절대 또는 상대 | 연구 단계 |
| ZFP | 10~30x | 고정 비트레이트 또는 정밀도 | 토모그래피에서 파일럿 |
| MGARD | 20~100x | 오차 제어 다해상도 | 연구 단계 |

**미해결 질문**: 각 측정 방식에서 어떤 오차 범위가 허용 가능한가?
이는 분야별 검증 연구가 필요하다.

### 3. FAIR 데이터 준수

DOE는 FAIR (Findable, Accessible, Interoperable, Reusable) 원칙
준수를 의무화하고 있다:

| 원칙 | 현재 상태 | 격차 |
|-----------|--------------|-----|
| **Findable** | 출판 데이터셋에 대한 DOI; Petrel 검색 | 시설 전체 카탈로그 없음; 비일관적 메타데이터 |
| **Accessible** | Globus 전송; Petrel 웹 포털 | 모든 데이터에 대한 표준화된 API 없음; 12개월 엠바고 |
| **Interoperable** | NeXus/HDF5 표준 | 빔라인 간 비일관적 NeXus 준수 |
| **Reusable** | NXprocess의 처리 출처 정보 | 불완전한 재현성 체인; 컨테이너화된 환경 없음 |

**새로운 해결책**:
- **Tiled** (BNL/NSLS-II)는 처음부터 FAIR 데이터를 위해 설계되었으며,
  HTTP를 통해 구조화된 데이터를 전체 출처 정보와 함께 제공한다. APS에서 테스트 중.
- **CRADLE** (Common Research Analytics and Data Lifecycle Environment)는
  전체 데이터 수명주기를 위한 분산 플랫폼으로, 이기종 데이터 관리,
  탄력적 확장, 접근 가능한 인터페이스를 제공한다.
- **온톨로지 프레임워크** (MDS-Onto, FAIRmaterials, FAIRLinked)가
  FAIR 원칙에 따라 방사광 데이터를 구조화하고 통합하기 위해 개발 중이다.

### 4. 소스에서의 데이터 축소

모든 것을 저장하고 나중에 처리하는 대신, 검출기나 빔라인에서
데이터를 축소한다:

| 전략 | 축소 배율 | 트레이드오프 |
|----------|-----------------|-----------|
| 관심 영역 크로핑 | 2~10x | 주변부 데이터 손실 |
| 검출기 내 압축 (Dectris) | 3~6x | 무손실, 트레이드오프 없음 |
| 스트리밍 피크 검출 | 100~1000x | 결정학에만 해당 |
| 실시간 사이노그램 조립 | 1x (재구성) | 리청킹 단계 제거 |
| AI 기반 희소 샘플링 | 5~20x | 검증된 모델 필요 |

### 5. 다중 시설 데이터 연합

BER 프로그램 사용자들은 종종 APS, NSLS-II, SSRL, LCLS의 데이터를 결합한다.
각 시설은 다음 사항이 서로 다르다:
- 파일 형식 및 규칙
- 메타데이터 스키마
- 접근 제어 시스템
- 데이터 카탈로그

**필요한 것**: 통합 인증(Globus 경유)과 표준화된 메타데이터(NeXus/Tiled 경유)를
갖춘, 여러 시설에 걸친 연합 데이터 접근 계층.

### 6. 엑사스케일 통합 (Aurora)

ALCF의 Aurora 슈퍼컴퓨터(2+ 엑사FLOPS)는 새로운 가능성을 열어준다:

| 역량 | APS에 대한 영향 |
|-----------|----------------|
| 엑사스케일 컴퓨팅 | 전체 볼륨 4D 토모그래피를 수 분 내에 완료 |
| AI 가속기 | 모든 APS 데이터에 대한 시설 규모 모델 학습 |
| 테라비트 APS-Aurora 링크 | 거의 제로 지연 데이터 전송 |
| 통합 메모리 | 단일 노드 RAM보다 큰 데이터셋 처리 |

**과제**: Aurora의 Intel GPU 아키텍처(SYCL/oneAPI)에 맞게 소프트웨어를
적응시켜야 하며, TomocuPy와 같은 CUDA 기반 코드의 포팅이 필요하다.

## 요약: 강점, 약점, 기회

### 강점
- 세계 최고 수준의 자동화된 파이프라인 (Bluesky + Globus Flows)
- ALCF를 통한 대규모 HPC 자원 (Polaris, Aurora)
- 명확한 보존 정책을 갖춘 검증된 계층형 스토리지 모델
- 강력한 커뮤니티 도구 (TomoPy, TomocuPy, MAPS, Bluesky)

### 약점
- 취득과 처리 사이의 HDF5 리청킹 병목
- 시설 전체에 걸친 제한적인 메타데이터 검색 및 탐색
- 재처리 캠페인에서의 HPSS 리콜 지연
- 빔라인 간 비일관적인 NeXus 준수

### 기회
- 실시간 실험 조종을 위한 엣지 AI
- 스토리지 비용 관리를 위한 손실 압축
- 원격 협업을 위한 Zarr/kerchunk 기반 클라우드 하이브리드
- 이전에 불가능했던 분석을 위한 Aurora 엑사스케일

### 위협
- 예산보다 빠르게 증가하는 스토리지 비용
- 소프트웨어 포팅 부담 (CUDA → SYCL for Aurora)
- 인력 병목 (빔라인당 소수의 데이터 엔지니어)
- 다중 모드, 다중 시설 실험의 복잡성 증가
