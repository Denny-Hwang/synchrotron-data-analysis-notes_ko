# TomoPy 대 TomocuPy -- 비교

## TomoPy의 강점

- **알고리즘 폭** -- 해석적 (gridrec, FBP), 반복적 (ART, SIRT, MLEM, OSEM) 및 정규화 (TV) 방법을 지원하여 광범위한 재구성 문제를 다룹니다.
- **성숙한 생태계** -- 광범위한 문서, 출판된 튜토리얼, TomoPy-cli 및 TomoBank 테스트 데이터셋과의 통합.
- **CPU 전용 운영** -- GPU 요구 없이 노트북에서 HPC 로그인 노드까지 모든 머신에서 실행됩니다.
- **플러그인 아키텍처** -- 사용자 코드 변경 없이 GPU 가속을 위해 ASTRA Toolbox 및 UFO 백엔드를 활성화할 수 있습니다.
- **큰 커뮤니티** -- 2014년 이래로 활발한 GitHub 이슈, 메일링 리스트 및 정기적인 릴리스.
- **빔라인 통합** -- APS, ALS, SLS 및 기타 방사광 형식을 위한 네이티브 리더.

## TomoPy의 약점

- **대규모 데이터셋에서의 속도** -- CPU 기반 gridrec은 2048^2 데이터셋에서 수분이 소요되는 반면, TomocuPy의 GPU에서는 약 25초.
- **C 확장 빌드 복잡성** -- 소스에서 컴파일하려면 작동하는 C 컴파일러가 필요하며 비표준 플랫폼에서 실패할 수 있습니다.
- **메모리 압력** -- 전체 사이노그램 스택을 한 번에 로드할 때 대규모 데이터셋이 사용 가능한 RAM을 초과할 수 있습니다.
- **스레딩 오버헤드** -- OpenMP 및 Python multiprocessing이 스케줄링 오버헤드를 도입하여 약 32코어 이상에서의 확장이 제한됩니다.

## TomoPy를 선택해야 하는 경우

1. 반복적 재구성 (SIRT, MLEM, TV)이 필요할 때.
2. NVIDIA GPU가 사용 불가능할 때.
3. 가장 넓은 알고리즘 선택과 확립된 커뮤니티 지원을 원할 때.
4. 프로토타이핑 중이며 광범위한 문서를 중시할 때.

## TomocuPy를 선택해야 하는 경우

1. 거의 실시간 해석적 재구성이 우선일 때.
2. 빔라인 워크스테이션에 NVIDIA GPU (Ampere 이상)가 사용 가능할 때.
3. APS 2-BM 또는 32-ID 데이터 관리 워크플로우 내에서 작업할 때.
4. 데이터셋이 단일 GPU 메모리에 맞거나 (또는 청크로 처리 가능할 때).

## 결합 워크플로우

APS에서의 실용적인 접근 방식은 데이터 수집 중 빠른 FBP 미리보기를 위해 TomocuPy를 사용하고, 출판 품질의 재구성을 위해 오프라인에서 TomoPy 반복적 솔버를 실행하는 것입니다.
