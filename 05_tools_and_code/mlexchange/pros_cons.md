# MLExchange -- 확장성 및 사용성 평가

## 강점

### 사용성

- **낮은 진입 장벽** -- 웹 GUI를 통해 빔라인 과학자가 Python 코드를 작성하거나 환경을 관리하지 않고도 ML 모델을 실행할 수 있습니다.
- **사전 구축된 모델 저장소** -- 일반적인 작업 (세그멘테이션, 노이즈 제거)이 기본 제공되어 첫 결과까지의 시간이 단축됩니다.
- **시각적 피드백** -- 학습 손실 곡선, 예측 오버레이 및 혼동 행렬이 브라우저에서 렌더링됩니다.
- **REST API** -- 고급 사용자가 프로그래밍 방식으로 워크플로우를 스크립팅하여 MLExchange를 자동화된 파이프라인에 통합할 수 있습니다.

### 확장성

- **마이크로서비스 설계** -- 서비스를 독립적으로 확장할 수 있으며, GPU 워커 추가 시 API 게이트웨이나 GUI 변경이 필요하지 않습니다.
- **다중 백엔드 컴퓨팅** -- 데이터셋 크기와 긴급성에 따라 로컬 GPU, NERSC HPC 또는 Kubernetes 클러스터로 작업을 라우팅할 수 있습니다.
- **컨테이너화된 모델** -- 각 모델이 자체 Docker 컨테이너에서 실행되어 의존성을 격리하고 재현성을 보장합니다.

### 전이 가능성

- **시설 비의존적** -- 원래 ALS용으로 구축되었지만 ALS 특화 가정 없이 설계되어 APS에 배포할 수 있습니다.
- **DLSIA 모델의 일반화** -- ALS 타이코그래피 또는 토모그래피 데이터로 학습된 아키텍처가 APS XRF 데이터에 성공적으로 미세 조정되었습니다.

## 약점

### 사용성

- **GUI에서 제한된 모델 커스터마이징** -- 고급 아키텍처 변경은 플랫폼 외부에서 Python 코드를 편집해야 합니다.
- **문서화 격차** -- API 레퍼런스 및 배포 가이드가 아직 발전 중이며, 일부 엔드포인트에 예제가 부족합니다.
- **노트북 통합 없음** -- Jupyter 사용자는 노트북 셀에서 MLExchange를 호출하는 대신 웹 GUI로 컨텍스트를 전환해야 합니다.

### 확장성

- **단일 노드 기본값** -- Docker Compose 배포는 단일 머신을 대상으로 하며, 다중 노드 Kubernetes 클러스터로 확장하려면 추가적인 운영 전문성이 필요합니다.
- **대용량 데이터셋 병목** -- 학습 전에 데이터셋을 Data Service에 업로드해야 하며, 수 GB의 토모그래피 볼륨의 경우 지연이 발생합니다.
- **스트리밍 학습 없음** -- 검출기에서 데이터가 도착하는 대로 모델을 학습할 수 없으며, 전체 데이터셋이 사전에 준비되어야 합니다.

### 운영

- **외부 서비스 의존성** -- MongoDB, RabbitMQ 및 컨테이너 런타임이 필요하여 배포 복잡성이 증가합니다.
- **인증 미성숙** -- 인증 서비스는 현재 기본 토큰 인증을 지원하며, 시설 ID 제공자 (예: Globus Auth, LDAP)와의 통합이 진행 중입니다.

## APS 도입을 위한 권장 사항

1. 영구 GPU 노드가 있는 APS Kubernetes 클러스터에 배포합니다.
2. 중복 업로드를 피하기 위해 Data Service를 APS Globus 데이터 패브릭과 통합합니다.
3. 사용자가 노트북에서 작업을 제출할 수 있도록 Jupyter 확장 또는 `mlexchange-client` Python 패키지를 추가합니다.
4. MLExchange 팀과 협력하여 Globus Auth / APS LDAP 지원을 추가합니다.
5. APS 특화 모델 템플릿 (XRF 세그멘테이션, 토모그래피 노이즈 제거)을 모델 저장소에 기여합니다.
